## ADDED Requirements

### Requirement: Frontmatter defines a Python generator instead of Rust algorithm

The `.md` file's frontmatter SHALL contain a `generator` field (YAML block scalar) holding a Python script that acts as a reference solution. This script SHALL read random inputs from stdin (generated by Rust WASM) and print the expected output to stdout. The frontmatter SHALL also contain a `params` object (YAML) specifying the random input parameter types and a `starter_code` field with the student template. No TOML file SHALL be used to store this data.

#### Scenario: ChallengeView reads generator from frontmatter

- **WHEN** a challenge page loads
- **THEN** `ChallengeView` reads `frontmatter.generator`, `frontmatter.params`, and `frontmatter.starter_code` without loading any external file

#### Scenario: Missing generator field causes visible error

- **WHEN** a challenge page loads and `frontmatter.generator` is undefined or empty
- **THEN** the challenge page displays an error state indicating the generator is missing

### Requirement: Rust WASM generates random inputs only

The `generate_challenge(params_json, count)` WASM function SHALL accept `params_json: &str` (JSON-serialized params object from frontmatter) and `count: usize` (number of testcases). It SHALL deserialize the params, generate `count` random input strings (one per testcase), and return an object containing only `inputs: Vec<String>`. It SHALL NOT compute `expected_output`. Param values SHALL be joined in the key order of the JSON object with newline separators to form each input string. `indexmap::IndexMap` SHALL be used to preserve key order.

#### Scenario: generate_challenge returns inputs in param order

- **WHEN** `generate_challenge` is called with params JSON `{"plaintext": {...}, "shift": {...}}`
- **THEN** each input string is `"{plaintext_value}\n{shift_value}"` in that key order

#### Scenario: generate_challenge respects count parameter

- **WHEN** `generate_challenge` is called with `count = 5`
- **THEN** the returned `inputs` array contains exactly 5 strings

### Requirement: Pyodide Worker executes generator to produce expected outputs

The Pyodide Worker SHALL handle a `generate` message containing `generatorCode: string` and `inputs: string[]`. For each input, it SHALL execute `generatorCode` with that input as stdin and capture stdout as `expected_output`. It SHALL respond with a `generate_complete` message containing `testcases: Array<{ input: string; expected_output: string }>`. If any generator execution throws an error, the worker SHALL include an `error` field in the response for that entry.

#### Scenario: Worker produces correct testcases from generator

- **WHEN** the worker receives a `generate` message with a valid generator script and inputs
- **THEN** it responds with `generate_complete` containing testcases where each `expected_output` is the stdout of running the generator with the corresponding input as stdin

#### Scenario: Worker reports generator errors without crashing

- **WHEN** the generator script throws a Python exception on a given input
- **THEN** the worker includes an `error` field for that testcase and continues processing remaining inputs

### Requirement: ChallengeView orchestrates two-phase testcase generation

`ChallengeView` SHALL execute testcase generation in two sequential phases on mount:
1. Call WASM `generate_challenge(params_json, count)` to obtain `{ inputs }`
2. Post a `generate` message to the Pyodide Worker with `generatorCode` (from frontmatter) and `inputs`, then await `generate_complete`

The challenge store SHALL be updated only after both phases complete. The `GeneratedChallenge` type in the store SHALL NOT include a `description` field.

#### Scenario: Challenge loads with complete testcases

- **WHEN** a user opens a challenge page
- **THEN** the left panel shows the markdown description immediately from `<Content />`, the editor loads `starter_code` from frontmatter, and testcases are available for running after both generation phases complete

#### Scenario: isGenerating skeleton shown during generation

- **WHEN** either WASM generation or Pyodide generator execution is in progress
- **THEN** the left panel shows a skeleton loader instead of content

### Requirement: Rust algorithms directory is removed

The `challenge-generator/src/algorithms/` directory and `src/template.rs` SHALL be deleted. The Rust WASM binary SHALL NOT contain any cipher algorithm implementations. The `list_algorithms` and `parse_challenge_meta` WASM exports SHALL be removed. `indexmap::IndexMap` SHALL be used for `params` to preserve JSON key insertion order.

#### Scenario: WASM binary does not export algorithm functions

- **WHEN** the WASM module is loaded
- **THEN** `list_algorithms` and `parse_challenge_meta` are not available as exports

#### Scenario: Params preserve declaration order

- **WHEN** frontmatter params are declared as `plaintext` then `shift`
- **THEN** the generated input string always places `plaintext` value before `shift` value
